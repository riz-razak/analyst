## 3. The rotation cascade caveat (Bala — READ THIS)

This project uses Supabase **legacy JWT keys** (wrangler.toml notes: "Settings → API Keys (legacy anon + service_role) + JWT Keys (legacy secret)"). For legacy keys, the anon and service keys are **both derived from the project JWT secret**. Therefore:

> **Rotating the legacy anon key = rotating the JWT secret = also invalidates the service key AND signs out every active session** (45 live `auth_sessions` at last count).

## 4. Options (founder picks ONE)

**Option A — Migrate to new Supabase API keys (recommended, clean).**
Supabase's new key system issues a publishable key (`sb_publishable_…`, replaces anon) and secret key (`sb_secret_…`, replaces service) that **rotate independently** without touching sessions. Steps: enable new API keys in the dashboard → set Worker secrets `SUPABASE_ANON_KEY` (=publishable) + `SUPABASE_SERVICE_KEY` (=secret) → remove fallbacks (§7) → deploy → disable legacy keys. Future rotations are then one-click, no cascade. Best long-term, also de-risks the millions-scale track.

**Option B — Minimal hygiene now, defer rotation (low-disruption).**
Treat the anon key as the public client key it is. Don't rotate (avoid the cascade). Just: confirm the Worker secret `SUPABASE_ANON_KEY` is set → remove the hardcoded fallbacks (§7) so the key lives only in the secret → deploy → retire the 3 orphan files. Pair with an RLS coverage audit (§5). Rotation happens later, folded into Option A.

**Option C — Full legacy rotate now (most disruptive).**
Rotate the JWT secret in Supabase. Invalidates anon + service + all sessions. Requires coordinated re-set of `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_KEY`, `SUPABASE_JWT_SECRET` on the Worker (and `yan-auth` Worker if it shares the secret) in the same window, then deploy. Everyone re-logs-in. Only in a planned maintenance window with a clear reason.

**Bala recommendation:** **Option B now** (kills the live fallback + git-forward exposure with zero disruption), **schedule Option A** as the proper fix during the Yan super-app/scale work. Avoid C unless we believe the key was actively abused.
